# 핵심 기능정의서

기준 날짜: `2026-03-08`

## 1. 문서 목적
이 문서는 `한국형 모바일 청첩장 서비스`의 P0 범위를 개발 가능한 수준으로 고정하기 위한 핵심 기능정의서다.

이 문서의 목적은 아래 3가지다.
- 무엇을 먼저 만들지 합의한다.
- 어떤 동작이 완료 상태인지 명확히 한다.
- 디자인/개발/운영 판단 기준을 동일하게 맞춘다.

## 2. 제품 정의
이 제품은 `웨딩화보를 가장 깔끔하고 세련되게 보여주고, 이름 + 사진 세트만으로 90초 안에 추천 preset 기반 모바일 청첩장 초안을 확인하게 만드는 서비스`다.

핵심 가치는 아래 순서로 평가받는다.
1. 첫 화면이 예쁜가
2. 사진이 살아나는가
3. 복잡하지 않고 바로 써볼 수 있는가
4. 발행 후 하객 대응이 가능한가

## 3. 목표 사용자
### 3.1 제작자
- 모바일 청첩장을 빠르게 만들고 싶은 예비부부
- 복잡한 템플릿 선택보다 사진 중심 결과물을 먼저 보고 싶은 사용자
- 준비된 sample preset에 자기 데이터를 갈아끼운 초안을 먼저 본 뒤 수정하고 싶은 사용자

### 3.2 하객
- 모바일 환경에서 청첩장을 열람하는 사용자
- 길찾기, RSVP, 방명록, 계좌 복사 같은 실제 행동이 필요한 사용자

## 4. P0 범위
P0에 포함되는 핵심 기능은 아래다.
- 랜딩 페이지
- Quick Draft
- sample preset 추천 / 전환
- 카드 뷰어
- 에디터
- 발행 / 비공개 미리보기 / 기본 통계

P0에 반드시 포함해야 하는 공개 카드 기능은 아래다.
- `main-parallax-screen`
- 갤러리
- 예식일시
- 예식장소 / 길찾기
- RSVP
- 방명록
- 계좌 복사
- 공유(OG / 카카오)
- 기본 통계

## 5. 비핵심 범위
아래는 P0에서 약하게 다루거나 제외한다.
- 장황한 회사 소개형 랜딩
- 초반 가격제 복잡도
- 초안 이전의 상세 정보 대량 입력
- 초안 이전의 긴 sample preset 비교 탐색
- 템플릿 쇼핑몰처럼 보이는 다량의 카드 카탈로그
- 관리자 기능 전체
- 정산, 쿠폰, 고급 멀티에디터 기능

## 6. 핵심 사용자 흐름
1. 사용자가 랜딩에 진입한다.
2. 랜딩에서 `Quick Draft 시작` CTA를 누른다.
3. 신랑/신부 이름과 사진 세트를 업로드한다.
4. 시스템이 추천 preset을 자동 적용해 초안을 생성한다.
5. 사용자는 비공개 미리보기에서 첫 화면과 전체 카드 느낌을 확인한다.
6. 필요하면 다른 sample preset으로 즉시 바꿔본다.
7. 필요하면 에디터에서 섹션 내용을 수정하거나 숨김 처리한다.
8. 카드를 발행한다.
9. 하객은 공개 카드에서 열람, RSVP, 길찾기, 방명록, 계좌 복사, 공유를 수행한다.

## 7. 기능 정의
### 7.1 랜딩 페이지
#### 목적
- 서비스 성격을 3초 안에 이해시킨다.
- `사진 넣으면 준비된 sample preset에 바로 합성된 결과를 볼 수 있다`는 메시지를 분명히 전달한다.
- 첫 CTA를 Quick Draft 시작으로 집중시킨다.

#### 필수 구성
- 서비스 한 줄 설명
- Quick Draft 시작 CTA
- 최소 수준의 sample preset 미리보기 노출
- 발행 후 가능한 기능 요약: RSVP, 방명록, 공유, 계좌 복사

#### 동작 규칙
- 첫 화면에서 과도한 입력 폼을 노출하지 않는다.
- 첫 화면에서 기능 카탈로그형 리스트를 길게 나열하지 않는다.
- 모바일 기준으로 첫 스크린에서 핵심 가치와 CTA가 보여야 한다.

#### 수집 이벤트
- `view_landing`
- `click_start_create`

#### 완료 기준
- 사용자는 첫 스크린만 보고도 서비스 목적을 이해할 수 있어야 한다.
- 첫 CTA는 Quick Draft 시작 버튼이어야 한다.
- 모바일에서 CTA가 접히지 않아야 한다.

### 7.2 Quick Draft
#### 목적
- 사용자가 `사진 세트 + 이름`만으로 추천 preset에 합성된 카드 첫인상을 즉시 판단하게 한다.

#### 입력 항목
필수 입력:
- 신랑 이름
- 신부 이름
- 메인 사진 1장
- 갤러리 사진 4장 이상, 최대 40장

선택 입력:
- 중간 사진 1장

#### sample preset 정책
- 초안 보기 전에 sample preset 선택을 강제하지 않는다.
- 시스템은 Quick Draft 생성 시 `추천 preset 1개`를 자동 적용한다.
- 추천 결과만 보여주고 끝내지 않는다.
- 첫 미리보기 직후 사용자는 `3~5개`의 준비된 sample preset 중 다른 것을 즉시 바꿔볼 수 있어야 한다.
- sample preset 전환 시 `이름`, `사진`, `하객 응대 데이터`는 유지되어야 한다.
- sample preset은 아래를 포함할 수 있다.
- 레이아웃
- 타이포
- 색상 토큰
- 소개문구 기본값
- 섹션 기본 노출 규칙
- 애니메이션
- 음악

#### 검증 규칙
- 메인 사진이 없으면 초안을 생성할 수 없다.
- 갤러리 사진은 최소 4장 이상이어야 한다.
- 갤러리 사진은 최대 40장까지만 업로드할 수 있다.
- 41번째 업로드 시 즉시 차단해야 한다.
- 차단 문구는 아래 문구를 그대로 사용한다.
`갤러리 사진은 최대 40장까지 업로드할 수 있어요.`

#### 자동 구성 규칙
- 중간 사진이 별도 업로드되지 않으면 갤러리 사진 1장을 자동 할당한다.
- Quick Draft 완료 직후 아래를 자동 생성해야 한다.
- `추천 preset` 적용
- `main-parallax-screen`
- 기본 갤러리 섹션
- 중간 섹션
- RSVP 섹션 기본 생성 및 기본 노출
- 비공개 미리보기용 draft 카드
- sample preset 교체 가능한 미리보기 상태

#### UX 원칙
- 초안 보기 전에 날짜, 시간, 장소, 계좌, 공지 입력을 강제하지 않는다.
- 초안 보기 전에 복잡한 옵션 설정을 요구하지 않는다.
- 초안 보기 전에 긴 sample preset 탐색을 강제하지 않는다.
- 첫 입력부터 첫 미리보기까지 `90초 이내`를 목표로 한다.

#### 시스템 동작
- 초안 생성 시 draft 상태 카드가 생성되어야 한다.
- 초안 생성 시 `presetId`, `presetVersion`, `presetSource`가 함께 결정되어야 한다.
- draft 카드는 소유자 token으로 비공개 미리보기가 가능해야 한다.
- Quick Draft 완료 후 즉시 카드 뷰어로 이동하거나 동일한 수준의 미리보기를 제공해야 한다.
- draft 미리보기에서는 다른 sample preset으로 즉시 전환할 수 있어야 한다.
- sample preset 전환 시 `default` 상태 섹션은 새 preset의 기본 payload, visibility, order 규칙으로 재계산될 수 있어야 한다.
- sample preset 전환 시 `edited` 또는 `hidden` 상태 섹션은 사용자 override로 유지되어야 한다.

#### 수집 이벤트
- `upload_photo_pack`
- `apply_recommended_preset`
- `auto_assign_photo_slots`
- `change_sample_preset`
- `view_main_parallax_screen`

#### 완료 기준
- 사용자는 필수 입력만으로 초안을 생성할 수 있어야 한다.
- 중간 사진 미입력 시 자동 대체가 동작해야 한다.
- draft 미리보기 URL 또는 동일한 접근 경로가 있어야 한다.
- 서버에서도 갤러리 40장 제한을 강제해야 한다.
- 추천 preset이 자동 적용되어야 한다.
- 첫 미리보기 이후 사용자는 입력값 재작성 없이 sample preset을 바꿔볼 수 있어야 한다.

### 7.3 카드 뷰어
#### 목적
- 제작자에게는 초안 품질 판단 도구여야 한다.
- 하객에게는 실제 참석 대응 도구여야 한다.

#### 필수 섹션
- `main-parallax-screen`
- 갤러리
- 예식일시
- 예식장소 / 길찾기
- RSVP
- 방명록
- 계좌 섹션

#### 7.3.1 Main Parallax Screen
##### 목적
- 카드의 첫인상을 결정하는 메인 히어로 섹션이다.

##### 요구사항
- 메인 사진이 가장 우선적으로 안정적으로 렌더되어야 한다.
- 신랑/신부 이름이 사진을 해치지 않는 방식으로 노출되어야 한다.
- 적용된 sample preset의 타이포와 색이 첫 화면에 반영되어야 한다.
- 사진이 카드 전체 무드를 결정해야 하며, 과한 UI 장식은 지양한다.

##### 완료 기준
- 첫 화면에서 사진과 이름이 명확히 보인다.
- 모바일 스크롤 첫 진입 시 깨짐 없이 자연스럽게 보인다.

#### 7.3.2 갤러리
##### 목적
- 웨딩 사진을 연속적으로 감상할 수 있게 한다.

##### 요구사항
- 기본 화면에서 썸네일 목록 또는 그리드가 보여야 한다.
- 사진 탭 시 전체화면 확대 모달이 열린다.
- 모달에서 이전/다음 이동이 가능해야 한다.
- 현재 인덱스 / 전체 수를 표시해야 한다.
- 닫기 버튼과 바깥 영역 탭으로 모달을 닫을 수 있어야 한다.

##### 참고 구현
- `ref/gallery-modal-demo.html`
- `ref/gallery-modal.js`
- `ref/gallery-modal.css`

##### 완료 기준
- 갤러리 사진은 업로드 순서대로 표시된다.
- 확대 모달 진입/이탈이 모바일에서 자연스럽다.
- 최소 4장, 최대 40장 정책이 뷰어에서도 일관되게 반영된다.

#### 7.3.3 예식일시
##### 목적
- 하객이 정확한 예식 일정을 이해하게 한다.

##### 요구사항
- 예식 날짜, 요일, 시간 정보가 노출되어야 한다.
- D-Day 표시는 선택 또는 기본 제공 가능하되 섹션 단위로 제어 가능해야 한다.

#### 7.3.4 예식장소 / 길찾기
##### 목적
- 하객이 장소를 쉽게 찾게 한다.

##### 요구사항
- 예식장 이름, 주소, 보조 설명을 입력할 수 있어야 한다.
- 길찾기 링크 또는 지도 연결 액션이 제공되어야 한다.
- 카카오 인앱 브라우저에서 동작 호환성을 우선 고려한다.

#### 7.3.5 RSVP
##### 목적
- 참석 여부를 수집한다.

##### 요구사항
- 하객이 모바일에서 간단히 응답할 수 있어야 한다.
- 제출 후 결과는 카드 소유자가 확인 가능해야 한다.

#### 7.3.6 방명록
##### 목적
- 축하 메시지를 남길 수 있게 한다.

##### 요구사항
- 공개 카드에서 메시지 작성이 가능해야 한다.
- 작성 결과는 카드에 반영되거나 소유자가 확인할 수 있어야 한다.

#### 7.3.7 계좌 복사
##### 목적
- 축의 전달 행동을 모바일에서 쉽게 만든다.

##### 요구사항
- 계좌 정보는 섹션 단위로 노출/숨김 가능해야 한다.
- 복사 버튼 클릭 시 즉시 복사 동작이 수행되어야 한다.

#### 7.3.8 공유
##### 목적
- 발행 후 카드 확산을 쉽게 만든다.

##### 요구사항
- OG 메타데이터를 제공해야 한다.
- 카카오 공유를 지원해야 한다.

#### 수집 이벤트
- `view_main_parallax_screen`
- `submit_rsvp`
- `write_guestbook`
- `copy_account`

#### 완료 기준
- 카드 뷰어는 모바일에서 우선적으로 읽기 쉬워야 한다.
- 공개 카드에서 RSVP, 방명록, 계좌 복사, 길찾기 중 핵심 행동이 실제로 수행 가능해야 한다.
- `main-parallax-screen` 과 갤러리는 P0에서 항상 렌더 가능 상태여야 한다.

### 7.4 에디터
#### 목적
- 사용자가 초안을 본 뒤 필요한 부분만 빠르게 수정하게 한다.

#### 핵심 원칙
- 모든 정보를 한 번에 입력하는 화면이 아니다.
- 미리보기와 실제 렌더가 최대한 동일해야 한다.
- 섹션은 독립 모듈이어야 한다.
- 섹션 순서 변경이 가능해야 한다.
- sample preset 교체가 가능해야 한다.
- 사용자 입력 데이터와 preset 표현 계층은 분리되어야 한다.
- 모바일에서도 편하게 사용할 수 있어야 한다.

#### Quick Draft 직후 기본 생성 섹션
- 예식일시
- 예식장소
- D-Day
- RSVP
- 방명록
- 부모님 성함
- 계좌 섹션
- 인터뷰/스토리

#### 섹션 상태
- `기본값`
- `수정됨`
- `숨김`

#### 요구사항
- 각 섹션은 수정 가능해야 한다.
- 각 섹션은 ON/OFF 또는 숨김 처리 가능해야 한다.
- 각 섹션은 순서 변경 가능해야 한다.
- 현재 적용된 sample preset을 표시해야 한다.
- 다른 sample preset으로 교체할 수 있어야 한다.
- 미리보기 데이터와 발행 데이터 구조는 최대한 동일해야 한다.
- `main-parallax-screen` 과 `gallery` 는 시스템 필수 섹션이라 P0에서 숨김/삭제할 수 없어야 한다.

#### 완료 기준
- Quick Draft 직후 생성된 기본 섹션을 사용자가 바로 수정할 수 있어야 한다.
- 섹션 순서 변경 결과가 미리보기와 실제 카드에 동일하게 반영되어야 한다.
- sample preset 변경 후에도 이름, 사진, 사용자 입력 데이터는 유지되어야 한다.

### 7.5 발행 / 비공개 미리보기 / 기본 통계
#### 목적
- 초안과 공개 상태를 명확히 분리하고, 발행 후 기본 운영 데이터를 확인하게 한다.

#### 상태 정의
- `draft`: 소유자 token으로만 비공개 미리보기 가능한 상태
- `published`: 공개 URL로 접근 가능한 상태

#### 요구사항
- 카드 조회 API와 카드 편집 API는 분리한다.
- draft와 published 상태를 명확하게 저장해야 한다.
- 소유자 token이 있으면 draft 카드도 비공개 미리보기가 가능해야 한다.
- card는 적용된 sample preset 정보를 함께 저장해야 한다.
- publish 동작 후 공개 카드 URL이 생성되어야 한다.
- 기본 통계는 최소한 아래 이벤트 기반으로 확인 가능해야 한다.
- 랜딩 진입
- Quick Draft 시작
- 초안 생성 관련 이벤트
- sample preset 추천 / 전환 이벤트
- 공개 카드 핵심 행동 이벤트

#### 완료 기준
- draft 상태에서는 비공개 접근만 가능해야 한다.
- published 상태에서는 공유 가능한 공개 접근이 가능해야 한다.
- 최소 수준의 핵심 행동 지표를 카드 단위 또는 서비스 단위로 확인할 수 있어야 한다.

## 10. 성공 기준
가장 중요한 KPI는 아래다.
- `TTFD(Time To First Draft)`
- Quick Draft 성공률
- 추천 preset 유지율
- 초안 -> 발행 전환율
- `view_main_parallax_screen` 이후 이탈률

제품 성공 판단은 아래 기준을 따른다.
- 고객이 빠르게 초안을 본다.
- 첫 화면이 예쁘다고 느낀다.
- 추천 preset 결과를 그대로 쓰거나, 적은 교체만으로 마음에 드는 초안을 찾는다.
- 발행까지 이탈하지 않는다.
- 하객이 RSVP / 길찾기 / 계좌 복사 같은 핵심 행동을 실제로 한다.
